# 面试讲解稿

这个项目是一个基于 RAG 与 MCP 思路实现的垂直领域知识库问答平台。用户可以上传文档，系统完成解析、切分、知识库构建和检索，然后把检索结果作为上下文交给模型生成回答。

## 三种模型模式

mock 模式是默认模式，不需要 API Key、Ollama、GPU 或本地模型。它会根据检索到的上下文返回模板化回答，主要用于验证文档上传、知识库构建、RAG 检索和 OpenWebUI 接口是否跑通。

API 模式通过 OpenAI-compatible Chat Completions 接口调用真实模型，例如 DeepSeek、OpenAI-compatible 服务或通义千问兼容接口。配置项包括 `API_MODEL_BASE_URL`、`API_MODEL_API_KEY` 和 `API_MODEL_NAME`。

Ollama 模式调用本机或 Docker 内部的 Ollama `/api/generate` 接口。配置项包括 `OLLAMA_BASE_URL` 和 `OLLAMA_MODEL`，例如 `qwen2.5:7b`。

## 为什么先用 mock 模式

实际开发中，一开始就依赖真实大模型会让问题定位变复杂：API Key、网络、本地模型、GPU、Ollama 服务都可能成为阻塞点。先用 mock 模式可以把工程主链路跑通，确认上传、解析、切分、检索、Prompt 拼接、接口格式和 OpenWebUI 接入都没问题，再替换成真实模型。

## OpenWebUI 如何接入

项目实现了两个 OpenAI-compatible 接口：

- `GET /v1/models` 返回模型名 `rag-mcp-knowledge-platform`；
- `POST /v1/chat/completions` 接收 OpenAI Chat Completions 风格请求。

OpenWebUI 发送聊天请求后，后端从 `messages` 中读取最后一条 `role=user` 的内容，作为用户问题进入 RAG Chain。RAG Chain 检索知识库片段，拼接 Prompt，再根据 `LLM_PROVIDER` 调用 mock、API 或 Ollama 返回回答。

## RAG 完整流程

完整流程包括：

1. 文档上传到 `data/raw_docs`；
2. SQLite 保存文档元数据；
3. Loader 解析 TXT、Markdown、PDF、DOCX；
4. Splitter 将长文本切分为 chunk；
5. Embedding 模块通过 Hugging Face sentence-transformers 生成文本向量；
6. Chroma Vector Store 持久化保存 chunk 和向量；
7. Retriever 根据用户问题取 Top-K 相关片段；
8. Prompt 模板把知识库内容和用户问题拼接；
9. LLM Client 生成回答；
10. 接口返回 answer 和 sources，便于追溯来源。

## 可优化方向

- 增加 FAISS、Milvus 等更多向量库实现；
- 增加重排序模型，提高 Top-K 检索质量；
- 增加用户、知识库和文档权限隔离；
- 增加增量更新、删除文档、去重和版本管理；
- 完善 MCP Server，接入更多外部工具和业务系统；
- 支持流式 token 输出、对话历史压缩和多轮问答；
- 增加前端管理界面和 OpenWebUI 使用截图。

## MCP 工具调用模块如何设计？

需要 MCP 是因为普通 RAG 问答主要围绕“用户问题 -> 检索 -> 生成回答”这一条链路，而实际智能体应用还需要调用外部工具，例如读取项目文档、查询数据库、获取知识库状态或直接执行检索。MCP Server 可以把这些能力以工具形式暴露出去，让客户端或智能体按需调用。

本项目的 MCP Server 当前实现为 HTTP-based MCP-like fallback 服务，工具函数集中在 `mcp_server/tools.py`，后续可以平滑迁移到官方 MCP SDK。它提供四个工具：`get_kb_stats`、`read_project_file`、`query_notes` 和 `search_kb`。

`get_kb_stats` 会读取 SQLite 中的 documents 和 chunks 表，并结合配置返回向量库类型、embedding backend 和 embedding 模型名。它不写死统计数字，而是读取当前真实知识库状态。

`read_project_file` 用于读取 README 或 docs 下的项目文档。它会把用户输入路径解析到项目根目录内，并禁止 `../../` 这类路径穿越；同时只允许 `.md`、`.txt`、`.json` 文件，并限制返回长度，避免一次返回过长内容。

`query_notes` 查询 SQLite 的 notes 表，支持按 title、content、tags 模糊匹配。初始化脚本会创建 notes 表，并插入 RAG、LangChain、MCP、OpenWebUI、AI 实习技术栈等示例笔记。

`search_kb` 复用现有 RAG 检索能力，也就是直接调用 vector_store 的相似度检索，而不是重新写一套检索逻辑。它返回结构化 chunks，包括 filename、chunk_id、score 和 text。

MCP 和普通 RAG 问答的区别在于：普通 RAG 输出的是自然语言回答，而 MCP 工具更偏结构化能力暴露。比如 `search_kb` 可以只返回检索结果，`get_kb_stats` 可以返回系统状态，`query_notes` 可以查数据库。这些工具可以被智能体编排，用于更复杂的任务流。
