# 项目名称：基于 RAG 与 MCP 的垂直领域智能知识库问答平台
# 适用场景：用于 AI 实习生岗位简历项目、GitHub 展示、面试讲解

你现在需要从零实现一个完整的 Python 工程项目，项目名称为：

rag-mcp-knowledge-platform

该项目是一个“基于 RAG 与 MCP 的垂直领域智能知识库问答平台”。项目需要具备工程化结构、可运行后端、文档上传、文档解析、文本切分、向量检索、RAG 问答、OpenWebUI 接入、MCP 工具调用、Docker Compose 一键部署、README 文档和测试用例。

本项目用于放入简历，因此不要只写 demo 代码。请按真实工程项目方式实现，要求代码结构清晰、注释适中、异常处理完整、README 可读、接口可测试、可以在本地运行和截图展示。

────────────────────────────────────────
一、项目最终目标
────────────────────────────────────────

实现一个支持私有文档知识库问答的 AI 应用平台，用户可以上传 PDF、Word、Markdown、TXT 等文件，系统自动完成文档解析、文本切分、向量化、向量库构建，并在用户提问时通过 RAG 检索相关片段，再调用大模型生成回答。同时系统需要提供 OpenAI-compatible API，使 OpenWebUI 能够作为前端聊天界面接入该后端。项目还需要实现一个简单 MCP Server，提供知识库统计、本地文件读取、SQLite 笔记查询等工具能力。

最终效果包括：

1. FastAPI 后端可以启动；
2. 可以上传文档；
3. 可以构建知识库；
4. 可以基于文档提问；
5. 回答中包含引用来源；
6. 可以通过 /v1/chat/completions 兼容 OpenWebUI；
7. 可以通过 Docker Compose 启动 backend、open-webui、ollama；
8. 有 MCP Server 示例工具；
9. 有 README、接口文档、测试案例；
10. 项目可以上传 GitHub 并写入简历。

四、后端 FastAPI 要求
────────────────────────────────────────

backend/app/main.py 负责创建 FastAPI 应用，加载所有 router。

需要注册以下路由：

1. /health
2. /api/documents
3. /api/knowledge
4. /api/chat

启动命令：

cd backend
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload

后端基础配置写在 backend/app/config.py，使用 pydantic-settings 读取环境变量。

五、接口设计
────────────────────────────────────────

1. 健康检查

GET /health

返回示例：

{
  "status": "ok",
  "app": "RAG MCP Knowledge Platform",
  "version": "0.1.0"
}

2. 上传文档

POST /api/documents/upload

请求：
multipart/form-data
file: PDF / DOCX / MD / TXT 文件

返回：

{
  "code": 0,
  "message": "document uploaded successfully",
  "data": {
    "doc_id": "uuid",
    "filename": "rag_intro.txt",
    "file_type": "txt",
    "saved_path": "data/raw_docs/uuid_rag_intro.txt"
  }
}

要求：
- 支持 .pdf .docx .md .txt
- 文件名需要安全处理
- 保存时加 uuid 前缀
- 超出支持类型返回 400
- 上传成功后将文档元数据写入 SQLite

3. 查看文档列表

GET /api/documents

返回：

{
  "code": 0,
  "data": [
    {
      "doc_id": "uuid",
      "filename": "rag_intro.txt",
      "file_type": "txt",
      "created_at": "2026-06-18 12:00:00"
    }
  ]
}

4. 构建知识库

POST /api/knowledge/build

请求：

{
  "doc_ids": ["uuid1", "uuid2"]
}

如果 doc_ids 为空，则构建全部已上传文档。

流程：
- 根据 doc_id 查找文档路径
- 解析文档
- 文本切分
- embedding
- 写入 Chroma 向量库
- SQLite 记录 chunk 数量

返回：

{
  "code": 0,
  "message": "knowledge base built successfully",
  "data": {
    "document_count": 2,
    "chunk_count": 38,
    "vector_store": "chroma"
  }
}

5. 知识库统计

GET /api/knowledge/stats

返回：

{
  "code": 0,
  "data": {
    "document_count": 3,
    "chunk_count": 120,
    "vector_store": "chroma",
    "embedding_model": "BAAI/bge-small-zh-v1.5"
  }
}

6. 普通 RAG 问答接口

POST /api/chat

请求：

{
  "query": "RAG 和普通大模型问答有什么区别？",
  "top_k": 5,
  "stream": false
}

返回：

{
  "code": 0,
  "message": "success",
  "data": {
    "answer": "RAG 会先检索外部知识库，再结合检索内容生成回答...",
    "sources": [
      {
        "doc_id": "uuid",
        "filename": "rag_intro.txt",
        "chunk_id": "uuid_chunk_0",
        "score": 0.82,
        "text": "RAG 是 Retrieval-Augmented Generation..."
      }
    ]
  }
}

要求：
- 如果知识库为空，返回明确错误或提示；
- 如果检索不到内容，回答“知识库中未找到相关信息”；
- sources 至少包含 filename、score、text；
- answer 不要编造来源；
- mock 模式下可以返回基于 context 的简单模板回答。

────────────────────────────────────────
六、RAG 模块详细实现要求
────────────────────────────────────────

1. loader.py

实现 DocumentLoader 类，支持：

load_file(file_path: str) -> LoadedDocument

LoadedDocument 包含：

doc_id
filename
file_type
content
metadata

不同文件类型处理：

.pdf 使用 pypdf.PdfReader
.docx 使用 python-docx
.md 使用普通文本读取
.txt 使用普通文本读取

要求：
- 统一使用 utf-8；
- 处理空文档；
- PDF 每页 metadata 中尽量保留 page；
- 出错时抛出明确异常；
- 不要让接口直接崩溃。

2. splitter.py

实现 TextSplitter 类。

推荐使用 langchain_text_splitters.RecursiveCharacterTextSplitter。

参数：
chunk_size = 800
chunk_overlap = 120

输出 Chunk 列表：

chunk_id
doc_id
text
metadata

chunk_id 格式：
{doc_id}_chunk_{index}

metadata 包含：
doc_id
filename
file_type
chunk_index

要求：
- 过滤过短文本；
- chunk text 去除首尾空白；
- 空文档返回空列表。

3. embeddings.py

实现 EmbeddingModel 类。

默认模型：
BAAI/bge-small-zh-v1.5

要求：
- 使用 sentence-transformers；
- 提供 embed_documents(texts: list[str])；
- 提供 embed_query(query: str)；
- 对中文支持较好；
- 如果模型下载失败，需要在 README 中说明可以换成 sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2；
- 可以在 .env 中配置 EMBEDDING_MODEL_NAME。

4. vector_store.py

优先实现 ChromaVectorStore 类。

功能：
- add_chunks(chunks)
- similarity_search(query, top_k)
- persist()
- count()

使用 chromadb 持久化目录 data/vector_db/chroma。

每个向量记录 metadata：
doc_id
filename
file_type
chunk_id
chunk_index

要求：
- 重复构建同一文档时避免大量重复。简单做法：先按 doc_id 删除旧 chunks，再写入新 chunks；
- 如果删除实现复杂，至少在 README 中说明当前版本重复构建会重复写入，并提供 clear 接口；
- 推荐实现 clear() 方法，用于清空向量库。

5. retriever.py

实现 Retriever 类。

retrieve(query: str, top_k: int) -> list[RetrievedChunk]

返回：
text
score
metadata

Chroma 的距离可能不是相似度，需要做简单转换：
score = 1 / (1 + distance)
或者直接返回 distance，并在字段中说明。

6. prompt.py

定义 RAG Prompt 模板：

你是一个专业的垂直领域知识库问答助手。
请严格根据【知识库内容】回答用户问题。
如果知识库内容不足以回答，请明确说明“知识库中未找到足够相关信息”，不要编造。
回答应清晰、有条理，必要时使用分点说明。
最后给出参考来源文件名。

【知识库内容】
{context}

【用户问题】
{question}

请生成回答：

7. chain.py

实现 RAGChain 类。

方法：

answer(query: str, top_k: int = 5) -> ChatResult

流程：
- 调 retriever 检索 Top-K；
- 组装 context；
- 构造 prompt；
- 调用 LLM；
- 返回 answer 和 sources。

要求：
- 检索结果为空时，不调用 LLM，直接返回“知识库中未找到相关信息”；
- sources 去重；
- sources 中保留 chunk 原文，方便前端展示；
- answer 中最好包含“参考来源”。
