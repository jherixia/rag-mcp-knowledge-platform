# 面试问答清单

## 1. 这个项目解决什么问题？

它解决私有文档知识库问答问题。传统大模型只能依赖训练参数回答，容易过时或编造；这个项目通过 RAG 先检索用户上传的私有文档，再把相关片段作为上下文交给模型回答，并返回 sources，提升可追溯性。

## 2. RAG 流程是什么？

文档上传、文档解析、文本切分、embedding 向量化、写入 Chroma、用户提问、query 向量化、Top-K 检索、Prompt 拼接、调用模型、返回 answer 和 sources。

## 3. 为什么要先做 mock 模式？

mock 模式可以先排除 API Key、网络、Ollama、本地模型、GPU 等不稳定因素，优先验证工程主链路：上传、解析、切分、检索、接口格式和 OpenWebUI 接入。等主流程稳定后，再切换 API 或 Ollama。

## 4. embedding 和 Chroma 怎么用？

embedding 使用 Hugging Face sentence-transformers，将文档 chunk 和 query 转为向量。Chroma 负责持久化保存 chunk 向量和 metadata。检索时用 query embedding 找 Top-K 相似 chunk，并将结果作为 RAG 上下文。

## 5. sources 为什么重要？

sources 让回答可追溯。用户可以看到回答依据来自哪个文件、哪个 chunk、相似度分数是多少，降低黑盒感，也方便排查错误回答。

## 6. OpenWebUI 是怎么接入的？

后端实现了 OpenAI-compatible API：`GET /v1/models` 返回模型列表，`POST /v1/chat/completions` 接收 OpenAI Chat Completions 格式请求。OpenWebUI 把后端当作 OpenAI 风格服务调用。

## 7. mock、API、Ollama 三种模式区别？

mock 模式不调用真实模型，用模板回答验证流程。API 模式调用 DeepSeek、OpenAI-compatible 或通义千问兼容接口。Ollama 模式调用本机或 Docker 内的 Ollama `/api/generate`。

## 8. 如何避免重复构建造成数据冗余？

构建或重建指定文档前，会先按 `doc_id` 删除 SQLite chunks 和向量库中旧向量，再写入新的 chunks 和 embeddings。

## 9. MCP 模块做了什么？

MCP 模块把项目能力封装成工具，包括知识库统计、项目文件读取、SQLite notes 查询和知识库检索。它和普通 RAG 问答不同，返回的是结构化工具结果，适合被智能体编排调用。

## 10. 这个项目还有什么不足？

官方 MCP SDK 还没接入，当前是 HTTP fallback；FAISS 只是预留接口；没有用户权限和多知识库隔离；没有前端管理页面；真实 embedding 首次运行需要下载模型。

## 11. 下一步怎么优化？

接入官方 MCP SDK，增加多知识库和权限管理，实现真正流式输出，增加 rerank 模型，补充前端管理页面，并完善 CI 和线上部署。
